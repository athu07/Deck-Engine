THE GOAL (never changes — for context)
------------------------------------
A tool where sales/tech fill in client context (client name,
industry, work type, deck phase, recipient) + optionally paste
a transcript, and the tool assembles a tailored PowerPoint by
picking slides from the master deck (and generating any missing
slides via template). Preview, then download.

I am new to development. Explain simply, one step at a time,
wait for my confirmation before moving on.

Design decision: CONTENT LIBRARY, not positional index.
Every slide = a standalone record (stable ID + text + tags).
Deck order does NOT matter. Slides are matched by content/tags.
The engine AUTO-TAGS slides (rules first, AI later for unclear ones).

Build order (one box fully working before the next):
  Box 0 — Explode deck into auto-tagged library        [DONE]
  Box 1 — Brain (matcher): context in -> slide_ids out [DONE, form-only/rules-only]
  Box 2 — Assembler: slide_ids -> new .pptx            [DONE]
  Box 3 — Form + web page the team uses                [DONE, runs locally]
  Box 4 — Deploy + v2 polish (transcript/AI, gap-fill,
          file-size cleanup, formatting, deploy)        <-- WE ARE HERE

Files in this folder:

WHAT WORKS (done & tested)
------------------------------------
Pipeline (all rules-only, no AI, no API cost):
  context -> matcher.py -> slide_ids -> assembler.py -> Tailored_Deck.pptx
  ...and a browser form (app.py) that does the whole thing + download.

Scripts:
  - stamp_ids.py     — stamps permanent J2W_ID into each slide's NOTES.
                       INSERT-SAFE: only un-ID'd slides get a new id
                       (max existing + 1); existing ids never renumber.
  - build_library.py — deck -> library.json (68 standalone records:
                       id + title/subtitle/text + pointer to slide).
  - tagger.py        — rule-based tags (kind/work_type/industry/function)
                       + per-tag confidence (AUTO/HUMAN) + keyword string.
                       vs registry: kind 88%, industry 67%, work_type 65%,
                       function 45%, keywords 100% captured.
  - ai_fallback.py   — OPTIONAL AI second opinion (OpenAI). Dry-run by
                       default; --apply to call the API. (See DECISIONS:
                       did NOT beat rules on this task — left in but unused.)
  - matcher.py       — Box 1 brain. Reads the registry's include_rule /
                       std_group / section to apply the Selection Rules.
                       Form-only for now (no transcript).
  - assembler.py     — Box 2. Keeps only chosen slides, in order, into a
                       NEW .pptx. Formatting preserved perfectly.
  - app.py           — Box 3. Flask form at http://127.0.0.1:5000 ->
                       preview -> download. Run with:  py app.py
  - secrets_loader.py + .env — API key kept out of code (.env is git-ignored).

Data files:
  - WORKING_COPY_Master_Deck.pptx — copy, all 68 slides ID-stamped.
  - library.json / tagged_library.json — the content library.
  - output/Tailored_Deck_*.pptx — generated decks.
Originals never modified (verified by byte-compare).

ADD LIBRARY SLIDES TO A DECK — "DECK TRAY" — DONE 23 June 2026
  Goal: from the Slide library, hand-pick any slide into the deck you're building.
  (NOTE: the Suggested-slides page already had an "Add slide" picker for all 133
  slides; this adds the same power from the nicer Library browse view.)
  HOW IT WORKS — a browser-side "deck tray" (localStorage key j2w_deck), no server
  session / no DB:
    - Library cards get a "+ Add to deck" button; a floating "Open deck (N)" pill
      shows the count and links to the deck. Cards already in the deck show a tick.
    - The deck tray stores {active, ctx{client/industry/phase/recipient/transcript/
      functions[]/work_types[]}, order[ids], buildId}.
    - Suggested-slides page (BUILD_BODY): on a NEW build it seeds the tray from the
      matcher picks + current form context (overwrites = fresh deck). Every reorder/
      add/remove persists the order back to the tray.
    - NEW route GET /deck re-opens the deck: rebuilds the slide list from the tray
      and refills the hidden form fields (client/phase/etc.) client-side, so
      Download/Review/Generate + meeting-logging all keep working unchanged.
    - buildId guard: re-opening the SAME build (e.g. browser Back) restores the deck
      as left (keeps library adds) instead of resetting to the matcher picks.
  UNCHANGED: matcher, assembler, meeting_log, /download, /review, /finalize — they
  still just receive the final slide-id list + context.
  LIMITATION (acceptable): if you start a deck COLD from the library (no build yet),
  there's no client name/phase context — /download then defaults client to "Client".
  Intended flow is build -> jump to library -> add -> Open deck. Manual adds also
  bypass matcher exclusions (you can deliberately add a divider/leader slide).
  Verified: routes /, /library, /deck, /meetings all 200; library renders add
  buttons + pill; /deck is in resume mode; /build renders with the tray vars.
